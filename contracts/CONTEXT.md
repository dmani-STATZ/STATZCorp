# Contracts Context

## 1. Purpose
A multi-company contract-workspace that owns the full lifecycle from contract header to CLINs, suppliers, financials, notes, reminders, dashboards, export-ready logs, and upload-ready SharePoint links. The current-state document (`contracts/CONTRACTS_APP_CURRENT_STATE.md`) outlines how Contracts stores header data, CLINs, suppliers, notes/reminders, GovAction tracking, payment history, folder-tracking stacks, and cross-tenant dashboards, so this app is the authoritative place for contract operations within the project.

## 2. App Identity
- **Django app name:** `contracts` (AppConfig `contracts.apps.ContractsConfig`, verbose name 'Contracts Management').
- **Filesystem path:** `contracts/` inside the repo root.
- **Role:** Core feature app; it exposes the main user portal for contract entry/review, plus admin tooling (companies, code tables, folder stacks) and supporting data (suppliers, buyers, IDIQ contracts) for the rest of the ecosystem.
- **Contextual hooks:** Most views expect `request.active_company` (see `contracts/views/mixins.py`), and the app decorates views with `STATZWeb.decorators.conditional_login_required` or `@login_required` to enforce authentication.

## 3. High-Level Responsibilities
- Host the canonical `Contract`/`Clin`/`Company` data model, including contract status flows, SharePoint document links, and CLIN-level financial splits (`ClinSplit` in `contracts/models.py`).
- Drive the operational UI: dashboards, contract management page, CLIN detail, contract log, folder tracking, finance audit, supplier search, and acknowledgements (`contracts/views/*.py` and `contracts/templates/contracts/`).
- Manage transactional data (notes, reminders, payment history, shipments, GovActions, acknowledgements) through forms, partials, and API endpoints so users can add contextual data without leaving the contract workspace (`contracts/forms.py`, `contracts/views/note_views.py`, `contracts/views/reminder_views.py`, `contracts/views/payment_history_views.py`). Reminder data is also accessible via a chrome-free popup window (`reminders_popup` view, `contracts/templates/contracts/reminders_popup.html`) opened from the reminder sidebar's pop-out button. The popup supports full create/edit/complete/delete/filter operations and redirects back to itself after all actions.
- Provide admin/superuser conveniences: code table CRUD (`contracts/views/code_table_views.py`), company CRUD plus branding controls (`contracts/views/company_views.py`), and supplier admin bulk imports (`contracts/views/admin_tools.py`).
- Expose structured APIs for UI widgets and integrators (`contracts/views/api_views.py`, `contracts/urls.py`), including select-option APIs, payment-history, IDIQ helpers, CLIN split JSON endpoints, shipments, and contract-day counts.

## 4. Key Files and What They Do
- `contracts/models.py`: defines the audit-friendly objects (`Contract`, `Clin`, `Note`, `Reminder`, `PaymentHistory`, `FolderTracking`, etc.), generic relation patterns, SharePoint link builder (`Contract.get_sharepoint_documents_url`), and helpers such as `FolderTracking.close_record`. Splits are stored per CLIN as `ClinSplit`; contract-level totals are computed on `Contract` (see Data Model).
- `contracts/forms.py`: styling mixin plus all ModelForms (CLIN, Supplier, Note, Reminder, AcknowledgementLetter, IDIQ, code tables) with custom cleaning logic (e.g., `ClinForm.clean` adjustments for NSN/supplier validation, `CompanyForm` membership sync and color validation).
- `contracts/urls.py`: wiring for dashboards, contract/CLIN CRUD, supplier screens, notes/reminders, folder tracking, finance/audit, API endpoints, IDIQ pages, superuser code tables, and admin tools.
- `contracts/views/contract_views.py`: the heart of contract CRUD, review toggles, expedite handling, contract search modal, and SharePoint/document UI for the management page; also runs CLIN/notes context for the main workspace.
- `contracts/views/clin_views.py` (+ `shipment_views.py`, `payment_history_views.py`, `acknowledgement_letter_views.py`): CLIN create/detail (field edits use the Transactions modal on `clin_detail.html`), shipment APIs, payment-history popup logic, and acknowledgement-letter generation/edit flows. - `contracts/views/supplier_views.py`: supplier list/detail/edit (reusing `suppliers.models`), inline updates (headers, addresses, notes, compliance), and supplier search/views tied into contract data for quick navigation.
- `contracts/views/contract_log_views.py`: paginated CLIN list used for exports, annotated with split totals and acknowledgement helper text.

## 5. Data Model / Domain Objects
- **Company (contracts/models.py):** stores tenant info plus branding/SharePoint URLs; enforces unique slug, supply defaults (`Company.get_default_company`), and has optional logo/colors used by templates.
- **AuditModel base class:** adds `created_by/modified_by` FKs (Django `User`) plus timestamp helpers used by `Contract`, `Clin`, `Reminder`, `Note`, etc.
- **Contract:** header-level data (`contract_number`, `pr_number`, `po_number`, `status`, `buyer`, `contract_type`, `scheduled dates`, `contract_value`, `plan_gross`, `special_payment_terms`, `company`). `pr_number` is a nullable `CharField` (max 50) for the Purchase Request number, copied from `ProcessContract` at finalization and editable through contract-edit workflows where enabled. Supplier is **not** stored on the contract header — use each `Clin.supplier` for supplier linkage. Has generic `Note` and `PaymentHistory` relations, and helper properties `total_split_value` / `total_split_paid` (runtime aggregates of all `ClinSplit` rows for CLINs on this contract) plus `get_sharepoint_documents_url`.
- **Clin:** belongs to a `Contract`, tracks supplier/NSN (`suppliers.models.Supplier`, `products.models.Nsn`), payment fields, due/ship dates, optional **`pod_date`** (`DateField`, null/blank, no `pod_date_late` flag — CLIN-level proof-of-delivery date, separate from per-shipment `ClinShipment.pod_date`), `special_payment_terms`, `GenericRelations` to notes/payment history, plus log fields and `ClinShipment` children. **`Clin.adjusted_gross`** (property): computed as `(item_value - quote_value)` minus the sum of `ContractFinanceLine.amount_billed` for that CLIN on **CLIN-level** finance lines only (`finance_lines` with `partial` NULL; shipment-scoped lines are excluded).
- **FinanceLineType:** seed lookup for finance line categories (`name`, `is_active`). Managed via seed migration — not exposed in UI in v1.
- **ContractFinanceLine:** CLIN- and partial-shipment ancillary cost records (`FK` → `Clin`, `related_name='finance_lines'`; optional nullable `FK` → `ClinShipment` as `partial` with `related_name='finance_lines'` on the shipment for shipment-scoped lines). `line_type` is free-text `CharField`, not an FK to `FinanceLineType`; optional `description`; `amount_billed`. `amount_paid`, `amount_remaining`, and `payment_status` are computed properties — no stored columns. CLIN-level lines keep `partial` NULL; lines tied to a partial shipment set `partial` to that `ClinShipment`. `Clin.adjusted_gross` sums `amount_billed` only for finance lines where `partial` is NULL so shipment-scoped lines do not alter the CLIN-level adjusted gross. Finance lines are **only** intended for the Finance Audit experience (`finance_audit.html` and its includes), not contract management or CLIN detail views.
- **FinanceLinePayment:** append-only payment events against a `ContractFinanceLine` (`FK` → `ContractFinanceLine`, `related_name='payments'`; `amount`, `payment_date`, optional `note`). Treat rows as append-only — do not update existing payment records in application logic.
- **PaymentHistory & Note:** generic relations to `Contract`, `Clin`, `ClinShipment`, and `IdiqContract` with audit fields; `PaymentHistory.clean` enforces payment-type validity per entity. `ClinShipment` uses entity type string `'clinshipment'` and partial payment types `partial_quote_value`, `partial_paid_amount`, `partial_item_value`, and `partial_wawf_payment`. For `entity_type='clinshipment'`, payment history writes update only the `ClinShipment` field directly (no parent `Clin` rollup and no `_sync_clin_ship_fields` call). `Note.note_tag` optionally distinguishes finance-only notes (`'finance'`) on the same `Contract` content type; see §4 finance notes bullet.
- **Add Partial Shipment modal (shared, 2026-05):** The Add Partial modal is a shared Bootstrap 5 modal included via `contracts/templates/contracts/includes/add_partial_modal.html`. It is used on both the Finance Audit page and the CLIN detail page (via `clin_shipments.html` in `mode="form"`). Modal JS lives in `contracts/static/contracts/js/add_partial_modal.js`. The modal POSTs to `/contracts/api/partials/add/` on both pages; on success it reloads the page (`window.location.reload()`). The auto-calc endpoint (`/contracts/api/partials/auto-calc/`) pre-fills Quote Value and Item Value when QTY is entered; the user may override. Manual edits are tracked via the `data-manually-edited` attribute on the quote/item inputs so auto-calc does not overwrite user input. The legacy `ClinShipments.addNewShipment()` inline-row pattern is retired for the primary entry point: the **Add New Shipment** button on the CLIN page opens this shared modal (class `js-open-add-partial`). `_sync_clin_ship_fields` runs after every successful partial create from `finance_line_views.add_partial_shipment` and after `shipment_views.create_shipment` / `update_shipment` / `delete_shipment`, keeping CLIN `ship_qty` and `ship_date` rollups aligned with child `ClinShipment` rows.
- **ClinShipment & ClinSplit:** `ClinShipment` supports multi-shipment tracking. `ClinSplit` (FK to `Clin`, `on_delete=CASCADE`, related_name `splits`) stores per-CLIN company splits (`company_name`, `split_value`, `split_paid`); `split_paid` is listed in `transactions.signals.TRACKED` for audit. Contract-level `total_split_value` / `total_split_paid` on `Contract` are computed aggregates, not stored columns. `ClinShipment` has an optional nullable `pod_date` (`DateField`) for Proof of Delivery; it is also listed in `transactions.signals.TRACKED` so edits create `Transaction` rows. **Financial fields (migration `0052`):** `quote_value`, `item_value`, `paid_amount`, and `wawf_payment` are nullable `DecimalField`s (`max_digits=19`, `decimal_places=2`). `quote_value` is supplier cost for this partial (conceptually `ship_qty × clin.price_per_unit`); `item_value` is contract value for this partial (`ship_qty × clin.unit_price`). Suggested amounts are exposed as read-only properties `auto_quote_value` and `auto_item_value`; the stored columns remain editable after any view/API auto-population. `paid_amount` is government payment for this partial; `wawf_payment` is customer/WAWF pay for this partial. Properties `finance_lines_for_audit` (this shipment’s `ContractFinanceLine` rows annotated with `amount_paid_sum` from payments) and `finance_billed_sum` (sum of `amount_billed` on those lines) support partial-shipment finance audit. `ClinShipment.ship_date` is nullable: `create_shipment` does not require a date at save time (client may omit or send `null`). In form mode, a **Today** button next to the ship-date input appears on new rows (`clin_shipments.js` `addNewShipment`) and on existing saved rows in `clin_shipments.html`. On CLIN detail, `clin_shipments.html` is included in `mode="form"`; the POD control is a button that calls `openTransactionsEditModal(window.clinShipmentContentTypeId, shipment.id, 'pod_date')` (content type id comes from `ClinDetailView` context as `clinshipment_content_type_id`). Legacy `shipment-modal` / `delete-confirmation-modal` markup was removed from that partial as unused. Shipment APIs in `shipment_views.py` stamp `created_by` / `modified_by` from `request.user`. After each successful `create_shipment`, `update_shipment`, or `delete_shipment`, `_sync_clin_ship_fields()` recomputes `Clin.ship_qty` (SUM of shipment `ship_qty`) and `Clin.ship_date` (MAX of non-null shipment `ship_date`) from all `ClinShipment` rows and saves those fields plus audit columns; `transactions` signals record CLIN field changes as usual. `complete_clin_shipping` still applies the final explicit CLIN totals unchanged. In `clin_shipments.html`, each saved shipment row carries audit fields as `data-*` attributes (comment, created by/on, modified by/on); the form-mode Actions column includes an info control that calls `ClinShipments.showAuditInfo` to open the Bootstrap modal `#shipment-audit-modal` (no extra audit API). In form mode, when `total_shipped == order_qty` (and `order_qty` is set), a **Complete Shipping** row appears in the table footer; it opens `#completeShippingModal` and posts to `complete_clin_shipping` to set CLIN-level `ship_date` / `ship_qty` (again, signals write `Transaction` rows). The row visibility is also updated client-side from `data-order-qty` on the section wrapper via `ClinShipments.updateTotalShipQty()`. `get_clin_details` in `gov_action_views.py` returns `is_partial` and `total_shipped` for the contract management CLIN card; `contract_management.html` shows an orange Bootstrap **Partial** badge next to Ship Qty when partially shipped (initial SSR and `fetchClinDetails` JS render).
- **ClinSplit.percentage (added Issue #15):** `DecimalField(max_digits=5, decimal_places=2, null=True, blank=True)`. Stores the intended split ratio for this company on this CLIN. Expected to be the same value across all CLINs on a contract for a given company name (enforced by convention, not DB constraint). When `update_clin_split` is called with `apply_to_all_clins=True`, all splits on the contract with the same `company_name` (case-insensitive) receive the same percentage via `.update()` — this is intentional and does not require audit trail. The `percentage` field is never auto-derived; it is always manually set.

- **recalc_splits view (Issue #15):** POST `contracts/<contract_pk>/splits/recalc/`. Loops all CLINs on the contract, computes `split_value = (percentage / 100) × adj_gross` per split, and writes the result to `ClinSplit.split_value`. If a CLIN has splits with no percentage set (all NULL), equal percentages are assumed (100/n each, first company absorbs rounding remainder). CLINs whose split percentages don't sum to ~100% are skipped and reported. Does NOT touch `split_paid`. Does NOT create audit notes (split_value changes go through normal `.save()` which fires transactions signals). Prefetches `finance_lines` to avoid N+1; computes `adjusted_gross` inline rather than using the `Clin.adjusted_gross` property (which bypasses prefetch cache).

- **Finance Audit Split Summary (Issue #15):** The Split Summary card now shows company percentage next to name, and each company row has an expand chevron that reveals a per-CLIN breakdown table (collapsed by default). Context adds `clin_splits_by_company` (dict: company_name → list of CLIN split rows) and `company_percentages` (dict: company_name → first non-null percentage).
- **IdiqContract & IdiqContractDetails:** capture umbrella IDIQ info plus allowed NSN/supplier combinations.
- **TrackerSchema & ContractRecord (dynamic contract tracker):** `TrackerSchema` is company-scoped, holds `name`, user-defined `columns` (JSON list; each column object may include optional `width_px` for persisted width), `column_order` (includes fixed ids `__contract__`, `__po__`, `__close__` interleaved with user column ids), **`system_col_widths`** (JSON object mapping those system column ids to pixel widths), and `is_active`. `ContractRecord` links an optional `Contract`, stores per-row `data` and `ui_state`, `status_sort_index`, and soft-close flags. Grid UI lives in `contracts/templates/contracts/dynamic_tracker.html` with views in `contracts/views/dynamic_tracker_views.py`. Column widths are saved with **POST** `/contracts/api/dynamic-tracker/<schema_id>/column-width/` (`contracts:api_update_column_width`): JSON body `{"column_id": "<user col id or __contract__|__po__|__close__>", "width_px": <int 40–800>}`. Resize handles on headers are **shown only when the tracker layout is unlocked** (hidden in locked mode). Toolbar jump-to search matches **contract number and PO number** after stripping non-alphanumeric characters for comparison (e.g. dashed and dashless contract forms match the same row); when several rows match, a **1 of N** counter and **◀ / ▶** buttons appear, and repeated **Go** / Enter cycles through matches in order.
- **Supplier, Buyer, ContractType, ClinType, SalesClass, SpecialPaymentTerms, CanceledReason, GovAction, FolderStack/FolderTracking, Reminder, AcknowledgementLetter, ClinAcknowledgment, Address:** all defined in `contracts/models.py` with FK ties back to `contracts.Contract` or suppliers and featured in forms/views for code tables, GovAction tabs, reminders sidebar, and folder-tracking UI.

## 13. Permissions / Security Considerations
- Views are decorated with `conditional_login_required` or `@login_required`; `ActiveCompanyQuerysetMixin` enforces company scoping and raises `PermissionDenied` without an active company (`contracts/views/mixins.py`).
- Notes: `delete_note` / `note_update` require creator or staff at the view layer (`contracts/views/note_views.py`). The notes **popup** tab panel hides **Edit** from non-creators (including staff on others’ notes); **Delete** remains creator or staff. Reminders: toggle completion checks reminder owner or staff (`contracts/views/reminder_views.py`).
- Superuser-only endpoints include `code_table_admin`, `company CRUD`, and `supplier_admin_tools` (`user_passes_test` wrappers + `user.is_superuser`).
- Some AJAX APIs are CSRF-exempt (shipments), but most JSON endpoints enforce POST/GET restrictions and login.
- Audit fields (`created_by`, `modified_by`) exist on the majority of models, so business logic must populate them (e.g., `ClinCreateView`).

## 14. Background Processing / Scheduled Work
- Management commands: `initialize_sequence_numbers` (re-syncs PO/TAB numbers), and `refresh_nsn_view` (now deprecated, only reports stats for the legacy view).
- No Celery tasks; background-like behavior includes `FolderTracking` exports and `ExportTiming` (which records timing so the UI can estimate export duration).
- Reminders are generated/read during requests via `context_processors.reminders_processor` (no periodic jobs).

## 15. Testing Coverage
- `contracts/tests.py` remains the default `TestCase` stub; no automated tests currently cover this app.
- Regression safety presently depends on manual verification of contract/CLIN workflows, so new features should add smoke/integration tests.

- 37 migrations exist (`contracts/migrations/0001_initial.py` through `0037_company_sharepoint_urls.py`), documenting the addition of payment history, folder stacks, supplier docs, GovActions, sharepoint URLs, etc.
- Many migrations adjust indexes (`0011_contract_plan_gross`, `0015_contract_contract_prime_idx`), restructure CLIN/Contract fields (`0030_contact_supplier_alter_supplier_contact`, `0034_contract_supplier_alter_clin_nsn_alter_clin_supplier_and_more`), simplify payment history (`0017_paymenthistory.py`), and add folder-tracking metadata (`0021_folderstack.py`, `0022_foldertracking_stack_id.py`).

## 17. Known Gaps / Ambiguities
- `contracts/views/acknowledgement_letter_views.py` references fields such as `recipient_name`/`recipient_address` that are missing from the `AcknowledgementLetter` model, so the view/template pair is likely stale.
- `contracts/views/note_views.api_add_note` is deprecated (URL retained for bookmarked links only; all creation goes through `add_note`). The endpoint references `request.content_type` (not set), so any legacy calls must pass `content_type_id` and `object_id` explicitly. Planned removal in a future cleanup.
- No automated tests exist, so assumptions about GovAction/log, reminders, or payment history have not been regression-tested.

**Gov Action AJAX:** Gov Action CRUD is fully AJAX. `gov_action_create` (POST `/contracts/contract/<id>/gov-action/create/`) and `gov_action_update` (POST `/contracts/gov-action/<pk>/update/`) both return a unified JSON payload via `_gov_action_to_json()` including raw field values (`action`, `request`, `initiated`) and display values (`action_display`, `request_display`, `initiated_display`, `date_submitted_display`, `date_closed_display`). The contract management template modal is dual-mode (Add/Edit), toggled by JS. The action column in each row uses a Bootstrap 5 kebab dropdown (`dropdown-menu-end`) containing Edit and Delete.
- Supplier admin CSV/XLSX import uses fuzzy matching, so renaming suppliers can break bulk updates if matching fails.

## CSS Architecture

The project no longer uses Tailwind in any form. The CSS refactor replaced all Tailwind with Bootstrap 5 and a custom three-file CSS architecture:

- `static/css/theme-vars.css` — CSS custom properties only (color tokens, brand vars, dark mode token overrides when `[data-bs-theme="dark"]` is on `<html>`, as set by `static/js/theme_toggle.js`). Hex values live here. Do not put layout or component styles here.
- `static/css/app-core.css` — layout, structure, and all component/button/modal styles. References `var()` tokens from `theme-vars.css`. New component classes go here.
- `static/css/utilities.css` — utility and helper classes.

`.modal-backdrop-overlay` in `app-core.css` is the project-wide standard class for all modal backdrop divs (`background: rgba(0,0,0,0.5)`). Do not use Tailwind `bg-gray-*/bg-opacity-*` combos for backdrops anywhere in the project — they are broken since Tailwind removal. This applies to all apps: contracts, processing, transactions, users, and root templates.

**When encountering Tailwind classes in templates:** replace with Bootstrap 5 equivalents or named classes in `app-core.css`. Do not leave Tailwind utility classes in place.

**Button pattern:** `.btn-outline-brand` in `app-core.css` is the standard outlined brand button. Use `.btn-outline-brand.btn-tinted` for a pill-style variant with a light `#eff6ff` background (e.g. the reminders pop-out button in `contract_base.html`).

**No wildcard button rules.** The former `button:not(...)` wildcard rules (for default chrome styling in light and dark mode) were removed. All `<button>` elements must have explicit Bootstrap 5 `btn` classes or a named component class defined in `app-core.css`. Never add a new wildcard `button:not(...)` rule.

## 18. Safe Modification Guidance for Future Developers / AI Agents
- Respect `request.active_company`. When querying models with a `company` FK, either filter manually or use `ActiveCompanyQuerysetMixin`; otherwise you risk leaking cross-tenant data.
- Changing contract/CLIN fields requires updates across `contracts/forms.py` (where applicable), the relevant templates/partials, `contracts/views/contract_views.py`, the Processing app when queue/process fields mirror Contract, search APIs, and `contracts/CONTRACTS_APP_CURRENT_STATE.md` so the documented workflow stays accurate.
- Renaming generic-related fields (`Note`, `PaymentHistory`) needs updates to the partials (`notes_list.html`), AJAX payloads (note modal/`api/add-note/`), and any exports that rely on `content_type_id`.
- Folder-tracking logic is tightly coupled with color constants (`FolderTracking.STACK_COLORS`, `folder_tracking.html`, and the Excel export helpers); change carefully when editing stack names or colors.
- Supplier/contact changes touch both `contracts/views/supplier_views.py` and the `suppliers` app models/forms; re-run or update `supplier_admin_tools` if file URLs or naming conventions change.
- Keep `contracts/CONTRACTS_APP_CURRENT_STATE.md` (and `Contracts Application.md`) in sync with code changes to preserve the documented expectations for this app’s UX.
